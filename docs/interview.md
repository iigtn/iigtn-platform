# Interview — 面接 Q&A（採用対策）

> 採用面接で iigtn.com について聞かれそうな質問と模範回答。
> ⚠️ これは **採用対策メモ** であり、READMEから分離しています。リポジトリ閲覧者の視線が散らない構成にするため。
>
> 各回答は **「弱点・限界も併記する」** スタイル。優等生回答だけだと逆に不信感を生むため。

---

## Q1. なぜ EC2 ではなくサーバレスにしたのですか？

> 個人ポートフォリオは可用性より長期維持コストと運用負荷ゼロを優先すべきと判断したためです。EC2 では月 600 円程度かかり、OS パッチ運用も発生します。サーバレスなら無風時 400 円程度（CloudWatch Logs 含む）に収まり、突発的なバズ流入にも自動でスケールします。
>
> ⚠️ ただし正直に言うと、**最初は EC2 + Nginx で考えていました**。Linux 力を見せたかったのですが、月数百円の常時課金が長期的に痛いと気付いて切り替えました。Linux 運用経験は別途検証用 EC2 をブログのネタとして残し、両方を訴求しています。

---

## Q2. Amplify や Vercel を使わなかった理由は？

> Amplify は内部で同じ AWS リソースを使いますが抽象化が強くインフラ層がブラックボックス化します。Vercel は Next.js を動かすには最適ですが、本プロジェクトの目的は **AWS 設計力の可視化** なので、手段が目的と矛盾します。

---

## Q3. CloudFront + S3 で OAI ではなく OAC を選んだ理由は？

> OAI は IAM Principal が `cloudfront.amazonaws.com` のレガシー機構で、SSE-KMS S3 や署名付き OAuth に対応しません。OAC は SigV4 ベースで、KMS 暗号化バケットや S3 以外のオリジンにも拡張可能です。AWS 公式も新規構築は OAC 推奨です。
>
> 実装時に Bucket Policy 側の `aws:SourceArn` 設定漏れで 403 を出した事故があり、OAC の「両側設定」の重要性を学びました。

---

## Q4. Terraform の State をどう管理していますか？

> S3 バックエンドにバージョニングと暗号化を有効にして保存し、DynamoDB でロックを取っています。dev/prod は別バケット・別ステートで分離し、cross-environment の事故を物理的に防ぐ設計です。実際 prod ステートに dev のつもりで apply しかけたことがあり、`prevent_destroy` が最後の砦として効きました。
>
> ⚠️ ただし module 構成は **3 回書き直して** 現在の 5 モジュール構成に落ち着いており、これが正解だとは言い切れません。案件規模で再考が必要だと思っています。

---

## Q5. Lambda のコールドスタートはどう対策していますか？

> 月数十回の問い合わせ用途なので Provisioned Concurrency はオーバースペックです。`arm64 + Node.js 20 + 軽量バンドル` で初回も 280ms 程度に抑えています。当初 Container Image で構築したら 1.2s 出たので、ZIP 形式に作り直して 4 倍改善した経緯を `docs/runbook.md` に残しています。

---

## Q6. CI/CD の権限設計は？

> GitHub Actions から AWS への認証は OIDC + AssumeRole で、静的キーは発行していません。Trust Policy で `repo:owner/repo:ref:refs/heads/main` まで絞り、prod デプロイは main からのみに制限しています。
>
> 実は **OIDC へ移行した直接の動機は、自分が GitHub Issue にデバッグログを貼った際 AWS_ACCESS_KEY が混じっていたことに気付いた事故** でした（runbook.md Case 8）。「Secret は人間が触れる場所に置かない」を徹底するきっかけです。

---

## Q7. DynamoDB と RDS、なぜ DynamoDB ですか？

> Lambda と RDS は同時実行数増加で接続枯渇するため、本番では RDS Proxy が必要になります。問い合わせフォームの PK 指定 Put / Query は DynamoDB に最適で、On-demand なら月 1 円から始められます。
>
> ⚠️ ただし **DynamoDB は書き味が辛い** とも感じています。JOIN がなく、要件が増えるたびに「Scan するか、GSI を 1 つ足すか」の判断を強いられる。データモデルが育つことを織り込んでいなかったのが反省で、規模が大きくなるなら S3 + Athena か RDS + Proxy に移行する判断もあり得ると思います。

---

## Q8. もしトラフィックが 100 倍になったら？

> S3 と CloudFront はそのまま自動スケール、Lambda も同時実行 1000 まで対応します。ボトルネックは DynamoDB On-demand スロットリングと SES quota。前者はキャパシティモード切替、後者は SES 上限引き上げ申請と SQS バッファリングで対応します。

---

## Q9. 障害時の確認順序を教えてください

> 外側から内側へ順に下ります。Synthetics → CloudFront → API Gateway → Lambda → DynamoDB → Route 53。Runbook を `docs/runbook.md` にまとめ、過去の事象も同じファイルに記録しています。直近では Case 1（CloudFront 403 OAC 漏れ）を学習目的で意図的に再現し、検知から復旧の流れを記録しました。
>
> ⚠️ 一方で、**深夜のアラートを翌朝まで気付かなかった事象** もありました（Case 7）。通知経路がメールのみだったのが原因で、現在は CRITICAL だけ別チャネルに分岐しています。技術より **通知設計の方が抜けやすい** と学びました。

## Q10. このシステムをクライアント案件に展開するとしたら？

> 構成のかなりの部分が士業・小規模企業のコーポレートサイト案件に流用可能だと考えています。clone してドメイン・配色を差し替え、microCMS を入れれば事務員が記事更新できます。月額 1,000〜3,000 円台が目安、Terraform で全構成が公開されているため引き継ぎも容易です。詳細な提案ストーリーは `docs/business.md` に記載しています。
>
> ⚠️ 一方で **会員登録 / EC / 法定保存系の要件には適合しない** とも書いており、案件次第で WordPress や別構成を提案する方が誠実だと考えています。

---

## Q11. このシステムの「いま一番悩んでいる」設計判断は？

> WAF の導入閾値です。平常時は過剰投資、攻撃を受けてからでは遅いため、**アラートベースで動的に WAF を有効化する** 仕組みを Lambda + EventBridge で組めないか検証中です。`docs/lessons.md` に Open Question として記録しています。
>
> 正直に言うと、**「攻撃を受けていないから後回しにしている」** という妥協が大きいです。本気で取り組むなら CDK で 1 日掛けて組めるはずですが、優先度を上げきれていません。

---

## Q12. コストが思ったより高かった経験は？

> 当初「Lambda は無料枠」と思い込んでいましたが、CloudWatch Logs の取り込み課金が月数十円〜百円程度発生することを見落としていました。Logs Retention を 30 日に明示的に縮めて対応しました。「無料」「ほぼ無料」と書く時に **副次課金** を含めて評価する習慣がつきました。

---

## Q13. （重要）まだ使いこなせていない領域は？

> 正直に言うと、いくつかあります：
>
> - **CloudWatch Logs Insights のクエリ** — 障害対応の最中に思い出せず手間取ったことがあり、`aws logs tail` で代替している場面がある
> - **Lambda のローカル実行 / モック** — `sam local` / `serverless-offline` で本番との挙動差にハマった経験があり、結局本番デプロイ + ログ確認で回している
> - **AWS WAF のルール設計** — マネージドルールに頼っており、カスタムルールでの細かい制御は経験不足
> - **Aurora Serverless v2 / RDS Proxy** — DynamoDB で済ませているため、RDS 系のサーバレス周りは Hands-on が浅い
> - **EventBridge Pipes / Step Functions の複雑なワークフロー** — シンプルな EventBridge Rules までは使えるが、Pipes / Step Functions の本格利用はまだ
>
> 「全部できます」と言うエンジニアは信用できないと思っているので、こういう **未習熟領域を可視化しておく** ことを意識しています。
