## 構成判断の背後にある経験

「壊れない構成」より「壊れてもすぐ気付ける構成」を優先したのは、過去に以下のような場面に出会ったからです。

- _[編集] 例：前職の現場で深夜に OS パッチ自動適用が失敗し、翌朝までサービスダウンに気付けなかった経験。_
- _[編集] 例：Terraform 化されていないコンソール手作業構成を引き継ぎ、設定差分の調査に 1 週間溶かした経験。_
- _[編集] 例：監視が CloudWatch Alarm だけで Runbook がない現場を見て、「自動化できないものはドキュメントで補う」という運用観を持った。_

つまりこの構成は技術カタログではなく、**「自分が次に運用するなら、過去の自分を救う構成」** を体現したものです。

## 設計の「揺らぎ」— 全部が合理的だったわけではない

> 設計判断はすべて綺麗に理由があるわけではない。**妥協・後回し・気分で決めた選択** も正直に書く。

- **最初は EC2 + Nginx で考えていた**。Linux 力を見せたい気持ちがあったが、月数百円の常時課金が長期的に痛いと気付いて切り替えた。**「正しい設計」よりも「自分が払い続けられる設計」** が決め手
- **Next.js を選んだのは半分は流行りに乗った結果**。Astro / Hugo の方が SSG 用途には軽いと後で気付いたが、移行コストを天秤にかけて据え置き中（[lessons.md](./lessons.md) §技術的敗北 参照）
- **CMS は最初から MDX に決めていたわけではない**。microCMS / Contentful / Sanity を 2 日触ってから「自分一人なら git で十分」と判断した。判断材料が増えた結果の選択
- **Terraform module 構成は 2 回作り直している**。最初は 10 個に細かく切ったが apply が遅くて 5 個に統合した
- **WAF を入れていないのは「正しいから」ではなく「まだ攻撃を受けていないから」**。動的 ON/OFF は構想止まりで、**現状は素のまま**

## 並行プロジェクトとの関係

本サイトは _[編集] 個人プロジェクト_ のインフラ基盤としても再利用可能な設計にしています。共通モジュール化することで、新しいプロジェクトを始めるたびにインフラを書き直す無駄を排除しています。

ただし**最初からそう設計したわけではない**。autoST など別プロジェクトを別アカウント・別 state で立てた後、共通する CloudFront / OIDC モジュールを抜き出して module 化した、という後付けの整理です。

## 「やらない」と決めたこと

派手だが今は不要、と判断した要素：

- **マルチリージョン active-active** — 個人サイトには過剰。SES 送信ドメインだけ将来検討
- **WAF の常時 ON** — 平常時は過剰投資。Bot 流入を観測してから動的に判断する方針（**未実装の構想**）
- **Lambda Provisioned Concurrency** — 月数十回のリクエストには不要。bundle 削減で対応

「採用したもの」より「採用しなかったもの」のほうが設計判断のシグナルになる、と考えています。
