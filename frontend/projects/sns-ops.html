SNS 運営フロー — 自社メディア連動の X アカウント運用

自社メディア (hojokin.iigtn.com / hojinka.com / iigtn.com) と連動した X (旧 Twitter) アカウントの運用フローを構築しています。アカウント設計からハッシュタグ戦略までを 60 ステップ超のスクリーンショットで記録し、将来的な投稿自動化（Lambda + EventBridge）を見据えた 再現可能な手順書として整備中。

A. 手動運用フェーズ（現在）

まず手動で運用フローを確立し、再現可能な手順書として整備するアプローチ。「自動化する前に、自動化すべき手順が 正しいか」を検証する目的です。

  1. アカウント設計 ― ペルソナ・bio・固定ツイート・ヘッダー画像を 3 メディアごとに設計
  2. プロフィール最適化 ― アバター 400px / ヘッダー light/dark の SVG → PNG レンダラを実装
  3. 初期投稿 ― 1 アカウントあたり 5〜10 件の固定型ポストを下書き → 公開予約
  4. 投稿スケジューリング ― X 公式 UI の予約投稿で平日朝 / 夜のスロットに割り当て
  5. ハッシュタグ戦略 ― 補助金 / 法人化 / AWS / Terraform / インフラ関連のタグで実験 → 反応率を計測

各ステップを 60 枚超のスクリーンショットで記録し、「次に運用を引き継ぐ人 (or 自分自身) が再現できる」 粒度に整備。これがそのまま自動化フェーズの仕様書になります。

B. 自動化フェーズ（次のステップ）

手動フローが安定したら、Lambda + EventBridge + X API v2 で投稿自動化に発展させる予定です。

  1. EventBridge ― 平日朝 / 昼 / 夜のスケジュールで Lambda を起動
  2. Lambda (post_scheduler) ― DynamoDB の posts_queue テーブルから次の予定投稿を取得
  3. Secrets Manager ― X API v2 の OAuth 2.0 Bearer Token / API Key を保管
  4. X API v2 (POST /tweets) ― 本文 + メディア + ハッシュタグを送信
  5. DynamoDB (posts_log) ― 投稿結果（tweet_id / 投稿時刻 / エラー有無）を記録
  6. CloudWatch Alarms ― 投稿失敗が続いたら SNS Email で通知

C. メディア連動の設計

各メディアの新規記事公開 → SNS 告知 → 反応率計測のループを回すため、メディア側の sitemap 更新を トリガに自動投稿候補を生成する設計を検討中です。

  • sitemap.xml の差分検知: Lambda が定期的に sitemap を取得して差分（新規 URL）を抽出
  • OGP メタの取得: 新規 URL から og:title / og:description / og:image を抽出して投稿テンプレートに流し込む
  • 投稿候補のキューイング: posts_queue テーブルに「24h 後 / 3d 後 / 1w 後」のリマインド枠で複数投稿を予約
  • 反応率計測: 投稿後 24h で imp / engage 数を取得して DynamoDB に記録、ハッシュタグ別の効果を継続評価

採用判断のポイント

項目採用理由
運用順序手動 → 自動化自動化する前に手順の正解を見つけることが重要。手動運用 = 仕様書化フェーズ
スケジューラEventBridgecron 表記で柔軟、Lambda と相性良し、追加コストほぼゼロ
状態管理DynamoDB On-Demand投稿頻度では十分、PITR 有効でロールバック可能
シークレットSecrets ManagerAPI キーをコードに置かない、ローテーション対応、IAM で参照権限を絞れる
APIX API v2投稿 / メディアアップロード / 計測 API が分離されており実装しやすい
記録媒体スクリーンショット 60+ 枚UI 変更時の差分追跡、後続の自動化 spec の根拠として残す

運用上の工夫

  • ヘッダー画像の SVG → PNG 自動レンダラ: frontend/assets/sns/_render_icon.py で SVG を 1500×500 PNG に変換
  • X strategy.md: 投稿頻度・カテゴリ・やらないこと（炎上対応 / 引用 RT 等）を明文化、運用ブレを防ぐ
  • 3 アカウント横断のトーン統一: 各メディアごとに bio・固定ツイートのトンマナを明示、サブアカ感を出さない

これからやる予定

  • 手動運用 30 日間の運用ログをまとめて、自動化要件定義に落とし込む
  • Lambda 投稿関数の PoC（X API v2 OAuth 2.0 + tweepy or 公式 SDK）
  • DynamoDB posts_queue / posts_log テーブル設計
  • Terraform モジュール化（他メディアでも流用できる形）
